# Desk Validation and QA

## Overview

Nothing the brain says becomes evidence until a command produces it this session, and the merge bar is the full suite green. Three fixture families that this repo colloquially calls "golden" pin three different things (scoring math, judgment quality, $0 replay) — conflating them produces wrong test edits and wrong panic when a check goes red.

## When to use / When NOT to use

| Situation | Go to |
|---|---|
| Deciding what evidence class a change needs, or reading this skill at all | **this skill** |
| Actually running eval steps 1–6 (emit/dispatch/record/verdict/rebaseline) | repo skill `run-eval` (mechanics) — this skill owns the WHY and the math, not the dispatch choreography |
| Deciding whether a change is DATA/PROMPT/FROZEN-CORE/DOCTRINE and what path it takes | `desk-change-control` (owns the classification, the Part-33 migration shape, the F-item lifecycle, fix-forward law) |

## 1. Three fixture families — the repo's naming blurs them, don't

| Family | Path | Pins | Who touches it | Regenerated by |
|---|---|---|---|---|
| **Scoring golden** | `fixtures/golden/` (`findings.json`, `ratings.json`, `status.json`, `anchors.json`, `scorecard.json`) | The deterministic CLI `run` path end-to-end — `gate.py`/`scoring.py`/`pipeline.py` math, via `tests/test_golden_integration.py` | Nobody, casually | **Only** a Part-33 migration with an independent hand computation in the commit message (precedent: commit `859ac35`, "regenerate golden fixtures once under v1.2 math" — hand-computed DMI/SMI/SDGI per (entity,indicator) and stated "Regenerated file matches exactly"). Route through `desk-change-control` §3 |
| **Eval golden set** | `fixtures/evals/cases/` + `fixtures/evals/baseline.json` | Brain-**prompt judgment quality** (F6, charter Part 24) — 18 cases, 8 extract-positive / 4 judge-positive / 2 thesis-positive / 4 frozen negatives | The `run-eval` procedure only | §4 below |
| **Recorded replay fixtures** | `fixtures/recorded/` (`extract-nvda.json`, `judge-nvda.json`, `ingest-merchant-gpu.json`) | $0 canned-answer replay of the live CLI paths (`--recorded` mode, demo mode) — nothing about quality, just determinism | Whoever intentionally changes the demo path | Hand-edit is fine — these are not golden in the quality sense, just recorded inputs |

None of these three is "the fixtures dir" — say which one you mean. A regeneration of one never substitutes for another (a golden-set rebaseline does not touch `fixtures/golden/`, and vice versa).

## 3. The hash-pin discipline

`tests/test_evals_baseline_pin.py::test_prompt_hashes_match_baseline` recomputes a per-seam SHA-256 over the canonical emitted `{system, schema, user}` bundle for a **fixed committed input** (`fixtures/evals/hash-input.json` — not the golden cases) and compares to `promptHashes` in `fixtures/evals/baseline.json`.

**Going red is the gate working, not a broken test.** The test's own docstring and failure message say this verbatim: *"PROMPT BUNDLE CHANGED — this is the F6 regression gate, not a broken test... Never hand-edit baseline.json."*

What flips it — broader than "I edited a prompt string":

- Any prompt-text edit in `gpu_agent/evals/emit.py`-reachable builders or the live CLI's matching `--emit-prompt` path (pinned equal by `tests/test_evals_hash.py::test_extract_emit_matches_live_cli_emit`, so the CLI side can't silently diverge from what's hashed).
- Pydantic answer-schema changes (`ExtractionResult`/`JudgmentResult`/`ThesisAnswer` — their `model_json_schema()` is part of the bundle).
- `registry/indicators.json` vocabulary data (ids/labels/sides/units feed `_extract_vocab`) — a "safe DATA edit" can still trip this.
- `docs/taxonomy.json` category ids, or CLI vocab-glue changes.

**The only sanctioned unlock:** the `run-eval` skill (re-dispatch brains + graders over the golden set under **current** prompts) → `gpu-agent eval rebaseline --runs <d1> <d2> <d3> --verdict <verdict.json>`. Hand-editing `fixtures/evals/baseline.json`, or any brain/grader answer, to make a check pass is **maintainer-confirmed forbidden** (2026-07-05) — the only lawful transformation on a saved answer is *transport normalization* (stripping markdown fences, taking the final complete JSON object when a model self-corrects mid-message), and every occurrence of that must be logged in the run notes as "content byte-identical."

## 4. Eval-v2 mechanics (charter Part 24, amended 2026-07-05 commit `9f891c9`)

Constants, verified live in `gpu_agent/evals/harness.py`:

```
BASELINE_SCHEMA_VERSION = 2
CRATER_DROP = 3          # a positive case craters at baseline-median - 3
HARD_CRATER_EXTRA = 2    # ...and hard-fails at baseline-median - 5
DISPERSION_LIMIT = 1.0   # replicate seam-mean range above this refuses to baseline
_EPS = 1e-9              # a value exactly on the bar PASSes
```

**Bar math.** For each seam: `ε = max(half-range of the 3 replicate seam means, one grading quantum)` where quantum = `1 / (number of positive cases in that seam)` — today extract `1/8=0.125`, judge `1/4=0.25`, thesis `1/2=0.5` (`seam_quanta()` derives this from the live case set, never hardcoded). `bar = seamMean − ε`; `hardBar = seamMean − 2ε`. A fresh run's seam value is scored `value >= bar − _EPS`.

**Crater prong** (independent of the seam mean): any single POSITIVE case scoring `≤ caseMedian − 3` fails the case regardless of how the seam averaged; `≤ caseMedian − 5` is a hard crater.

**Verdict ladder** (`evaluate_v2()`):

| Reports supplied | No fail | Any fail, no hard-fail | Any hard-fail |
|---|---|---|---|
| 1 (gate run) | `pass` | `marginal-fail` | `hard-fail` |
| 2 (the one sanctioned replication) | `pass` | `fail` (final — no third run, ever) | `fail` |

Plus `invalid-run` (grader miscalibrated on a negative — score above `max_score(seam)//2 = 4`; a seam with a baseline mean but zero scored positives; a baseline case scored in no run; or, for 2-report replications, prompt hashes differing between the two runs). Two edge states exist in code but are functionally historical now that the committed baseline is already schema v2: `bootstrap` (no baseline exists yet — first-ever eval run) and `no-comparison` (an existing baseline is schema v1 — the v1→v2 migration path).

**Current committed baseline** (verify: `.venv\Scripts\python -c "import json; b=json.load(open('fixtures/evals/baseline.json')); print(b['schemaVersion'], b['seamMeans'], b['epsilon'])"`), observed 2026-07-06 — unchanged since the F63 re-gate (`ef52790`, last commit to touch this file):

```
schemaVersion 2
seamMeans  {extract: 6.75, judge: 7.5833, thesis: 6.0}
epsilon    {extract: 0.3125, judge: 0.25, thesis: 0.5}
bars       {extract: 6.4375, judge: 7.3333, thesis: 5.50}
```

Its 3 replicate `runDirs` (`work/eval-f63-regate-2026-07-05/r1..r3`) do **not** exist in the root checkout — they live only in the retained worktree `.worktrees/f63-corroboration/work/` (`ls .worktrees/f63-corroboration/work` confirms `eval-f63-regate-2026-07-05/`). This is why the repo forbids `git clean` anywhere: the committed baseline's raw evidence is gitignored `work/` inside a worktree with no registry pointing at it (F76c, open).

**A subtlety worth knowing before parsing `verdict.json`:** the crater `value` field is the single-run total when 1 report was supplied but the two-run mean when 2 were — same key, different meaning depending on how many reports you handed `evaluate_v2`.

Full worked numeric walk-through (F62/F63 saga, the epsilon/bar arithmetic by hand) is in `references/eval-v2-worked-example.md`.

## 5. What counts as evidence here

- **A number is citable only if a command produced it THIS session.** Quoting a discovery doc, a prior run's printed number, or a doc's stale count is not evidence — re-run `--collect-only`, re-open `baseline.json`, re-run the pin.
- **Dispositions are pre-committed before results exist.** Every eval run-note in `docs/superpowers/eval-notes/` and `docs/superpowers/*-eval-run-notes.md` states the PASS/FAIL handling plan *before* `record-grade` executes. "Retry until green" is explicitly forbidden — the one time the bar itself was provably wrong (F63 failing twice against F62's high-draw incumbent), the fix was to rebuild the bar (eval-v2) as its own feature, not to force past the old one.
- **Every re-dispatch is logged with the verbatim violation text appended** — gate rejection of a fresh answer is signal that the *candidate prompt* is bad, never an eval bug, and never a reason to touch the answer by hand.
- **Raw run dirs are preserved forever** — `work/eval-*` and worktree-nested equivalents are the only record of what a replicate actually scored; the committed `baseline.json` is a summary of them, not a replacement.
- **"A cycle that isn't committed didn't happen"** (`CLAUDE.md`) — this applies to eval runs too: an eval PASS with no rebaseline commit is not a qualified prompt.
- **Carried negatives are a SESSION procedure, not code** — a known weak point. `build_grade_prompt()` embeds only the brain's verbatim USER prompt (never its SYSTEM) into the grade prompt, so when a run's negative-case grade prompts verify byte-identical to a prior run's (checked by the session, not asserted by any test), that prior run's grades (observed reproducing as `2/1/0/2` across the F62/F63/eval-v2 runs) may be carried over instead of re-dispatching graders. A SYSTEM-only prompt change legitimately leaves negatives byte-identical; a USER-prompt-affecting change re-arms them. Nothing in the harness verifies this byte-identity for you — get it wrong and you either burn graders on unchanged negatives or silently carry stale grades forward.

## 6. How to add a test — decision table

| You're doing this | Do this |
|---|---|
| Testing `gate.py`/`extractor.py`/`judge.py`/`sufficiency.py` behavior (a violation string, an acceptance path) | Follow the `_base(**over)` builder house pattern (below) — one behavior per test function, assert on a substring inside the violations list |
| Testing an `eval` CLI verb's plumbing (exit codes, stage-file contracts) | Follow `tests/test_cli_eval.py`'s pattern: `tmp_path` + `main([...])` calls, assert exit codes 0/1/2 precisely (2 = operator error/missing stage file, 1 = gate or regression failure, 0 = clean) |
| Widening judgment-quality coverage for a new brain failure archetype (a new corroboration wrinkle, a new negative you want frozen) | Add a case file to `fixtures/evals/cases/` with real `notes` (≥40 chars), a real `source`, and honest `checks`. This does **not** require an eval run or trip the hash pin (the pin hashes a separate fixed `hash-input.json`, not the case set) — but it must clear `tests/test_evals_fixture_health.py`'s census floors (≥5 extract / ≥4 judge / ≥4 thesis positives, ≥3 negatives spanning all 3 seams, ≥15 total) and re-emit cleanly. The new case's median won't enter `baseline.json` until the next 3-replicate rebaseline folds it in |
| Pinning that a scoring-math change produced the RIGHT numbers end-to-end | This is a `fixtures/golden/` regeneration — Part-33 migration only, independent hand computation required. Route to `desk-change-control` |
| Changing the CLI demo/`--recorded` path's canned inputs | Edit `fixtures/recorded/*` directly — no eval or golden implications |
| Testing report/brief rendering | Deterministic projection tests (pure function of a scorecard) — no LLM, no eval; the rendered brief is ungraded by design (§7) |

**House pattern 1 — gate/extractor/judge unit tests** (`tests/test_gate_finding.py`): build a base valid object via a `_base(**over)` helper, then override just the field under test:

```python
def _base(**over):
    data = {"id": "f", "statement": "s", "kind": "observed", ...}
    data.update(over)
    return Finding.model_validate(data)

def test_observed_with_value_is_invented_number():
    f = _base(kind="observed", value={"number": 5.0, "unit": "x"})
    assert any("invented value" in e for e in check_finding(f))
```

One behavior per test, assertion on a violation substring — never on exact string equality (violation wording changes; the category of violation is the contract).

**House pattern 2 — eval CLI plumbing tests** (`tests/test_cli_eval.py`): write minimal cases into `tmp_path`, call `main([...])` directly (no subprocess), assert the exit code AND the stderr text naming the missing prerequisite. `test_record_steps_missing_stage_input_exit_2` is the clean template for "operator forgot a step."

**House pattern 3 — golden integration** (`tests/test_golden_integration.py`): run the full CLI `run` verb against `fixtures/golden/`, diff `demandSupply` and index invariants against the committed `scorecard.json` (provenance popped before comparison — it's non-deterministic by design, timestamps).

**House pattern 4 — fixture health** (`tests/test_evals_fixture_health.py`): census floors, every case re-emits, every frozen `recordedAnswer` still produces its recorded `checks.gateOutcome` under **current** gates (this is what catches contract drift silently rotting the golden set — a case that used to gate-pass but now gate-fails under a newer rule is a real signal, not a fixture bug), `mustMention` holds on positive frozen answers, notes/source are substantive.

**Dead field, don't build on it:** `checks.citationsResolve` is declared (default `True`) on every case and set in every fixture, but no gate, harness, or test reads it — grep confirms only the pydantic field declaration and the fixtures themselves. Real citation-integrity coverage happens through the actual production gates inside `gate_brain_answer()`.

Deeper walk-through with more file excerpts: `references/adding-tests-house-patterns.md`.

## Common mistakes

- **Treating any fixture-related test failure as "the golden set is broken."** Identify which of the three families it is first (§1) — the fix path is completely different for each.
- **Trying to fix a red `test_evals_baseline_pin.py` by editing `baseline.json` or the test itself.** Both are forbidden; the only path is `run-eval` + `eval rebaseline`.
- **Treating a marginal eval verdict as meaningful on its own.** Given F73, a pass/fail margin inside ~0.4 (the documented noise floor) is a noise event until F73 lands — that's an argument for improving gate power, never a license to skip the gate.
- **Adding a golden-set case and assuming it must immediately trigger a full eval run.** It doesn't (§6) — it just needs to pass fixture-health and wait for the next natural rebaseline.
- **Assuming eval judge behavior mirrors live judgment.** Eval deliberately differs: `samples=1, resample_budget=0`, memory comes from the case's frozen `memoryText` (no store dependency) — this is a documented deviation (`gpu_agent/evals/emit.py` module docstring), not a bug to "fix" toward live's 3-sample aggregation.
- **Assuming `citationsResolve` gates anything.** It's declared, set everywhere, enforced nowhere (§6).
- **Quoting readme.md's suite-status numbers.** They are stale; run `--collect-only` yourself every time.
- **Believing the crater `value` field means the same thing in every `verdict.json`.** It's single-run total for 1 report, two-run mean for 2 (§4).
- **Forgetting that adding a new indicator/vocab label to `registry/indicators.json` is "just data."** It feeds `_extract_vocab` and can flip the hash pin — a "safe" data edit can still be a PROMPT-class change (owned by `desk-change-control`, but the QA symptom — the pin going red — lands here).
